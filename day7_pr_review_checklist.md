# Day 7 - PR 初审检查清单

## 目标

这份清单用于对一个 Pull Request 做第一轮审查。第一阶段不是替代资深 reviewer，而是帮助客户快速发现明显问题、缺失测试和高风险改动。

固定输出：

1. 本次改动摘要
2. 潜在 bug
3. 缺失测试
4. 兼容性风险
5. 安全或权限风险
6. 建议 reviewer 重点看的地方
7. 是否建议合并

## 1. 基本信息

需要记录：

- 仓库链接
- PR 链接
- PR 标题
- 作者
- base 分支
- head 分支
- 改动文件数
- 新增/删除行数
- CI 状态

## 2. 改动摘要

检查目标：

用 1-3 句话讲清楚这次 PR 做了什么。

好摘要：

- 本 PR 为登录页新增表单校验，并调整了错误提示文案。
- 本 PR 更新 GitHub Actions，将 Node 版本从 18 升级到 20。
- 本 PR 修复 pytest 中缺失依赖导致的 CI 失败。

坏摘要：

- 改了一些代码。
- 优化项目。
- 修复 bug。

## 3. 潜在 bug

重点检查：

- 是否有空值/未定义风险
- 是否有路径大小写问题
- 是否有异常未处理
- 是否有边界条件遗漏
- 是否有异步流程未等待
- 是否有配置名拼错
- 是否有测试环境和生产环境不一致

客户可读解释：

PR 初审最重要的是先找“可能合进去就出问题”的地方。

## 4. 缺失测试

重点检查：

- 是否改了业务逻辑但没有测试
- 是否修了 bug 但没有回归测试
- 是否新增功能但没有最小验证
- 是否只改了测试快照但没说明原因
- 是否 CI 没覆盖这次改动

建议输出：

- 建议补充一个覆盖正常路径的测试。
- 建议补充一个覆盖失败输入的测试。
- 建议补充一个回归测试，防止该 bug 再次出现。

## 5. 兼容性风险

重点检查：

- 是否改了公开 API
- 是否改了配置字段
- 是否升级依赖
- 是否改了 Node/Python 版本
- 是否改了数据库结构
- 是否改了构建产物路径
- 是否影响旧浏览器或旧系统

风险判断：

- 文档/注释改动通常低风险
- CI/workflow 改动通常低到中风险
- 依赖升级通常中风险
- API/数据库/权限改动通常高风险

## 6. 安全或权限风险

重点检查：

- 是否新增或修改 secrets
- 是否扩大 GitHub Actions 权限
- 是否打印 token/key
- 是否修改登录逻辑
- 是否修改支付逻辑
- 是否修改权限判断
- 是否处理用户个人信息
- 是否引入外部脚本或第三方 action

高风险提示：

涉及登录、支付、权限、数据库、生产部署、secrets 的 PR，不建议只靠 AI 通过，必须人工重点 review。

## 7. Reviewer 重点关注点

输出格式：

```text
建议 reviewer 重点看：
1. src/auth/login.ts 的权限判断是否正确。
2. .github/workflows/ci.yml 的 permissions 是否过大。
3. tests/login.test.ts 是否覆盖失败输入。
```

每条都要具体到文件、函数、配置或测试。

## 8. 是否建议合并

使用三个等级：

- 可以合并
- 暂缓合并
- 需要人工确认

可以合并：

- 低风险文档/模板改动
- CI 通过
- 没有明显缺失测试
- 没有安全或权限风险

暂缓合并：

- CI 未通过
- 缺少关键测试
- 有明显 bug 风险
- 依赖升级未说明

需要人工确认：

- 登录/支付/权限
- 数据库迁移
- secrets
- 生产部署
- 大范围重构

## 9. 人工复核点

AI 输出交付前必须确认：

- 是否只基于本次 diff
- 是否编造不存在文件
- 是否忽略 CI 失败
- 是否把低风险说成高风险
- 是否把高风险说成低风险
- 是否给出空泛建议
- 是否残留敏感信息

## 10. 合格 PR 初审报告标准

合格报告必须包含：

- 改动摘要
- 具体问题
- 具体文件或逻辑位置
- 测试建议
- 风险等级
- 是否建议合并
- 人工确认点
