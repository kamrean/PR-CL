# 最终服务蓝图

## 定位

这是一个从轻量人工服务起步、逐步向半自动执行层推进的 PR/CI 工程化服务仓库。

它解决的核心问题不是“帮客户再造一个聊天机器人”，而是把代码协作里最容易卡住的几步做得更可控：

- 仓库接手困难
- CI 红灯定位慢
- PR 风险判断慢
- GitHub Actions 基础薄弱

## 当前默认支持范围

当前主支持：

- GitHub
- GitHub Actions
- Node 项目
- Python 项目

以下内容默认按单评估，不作为标准承诺：

- GitLab CI
- Jenkins
- CircleCI
- Vercel / Netlify / Cloudflare Pages 平台级问题
- 多语言 monorepo
- 生产部署

## 目标客户

- 学生项目组
- 独立开发者
- 小型外包团队
- 小 SaaS 团队
- 旧仓库维护者

## 四类核心服务

### 1. 仓库体检

交付内容：

- 仓库体检报告
- 固定检查项结果
- 风险分级
- 修复优先级
- 下一步建议

固定检查项：

- README
- 依赖安装
- 测试命令
- CI 配置
- secrets 风险
- PR 模板
- 分支保护

建议价格：

- `499-999 元 / 仓库`

### 2. GitHub Actions 的 CI 红灯诊断

交付内容：

- 失败步骤
- 关键日志
- 根因判断
- 修复建议
- 验证命令
- 风险等级
- 是否适合自动修复

建议价格：

- `99-299 元 / 次`
- `999-2999 元 / 月` 小团队托管

### 3. PR 初审

交付内容：

- 改动摘要
- 潜在 bug
- 缺失测试
- 兼容性风险
- 安全或权限风险
- reviewer 重点关注项
- 是否建议合并

建议价格：

- `99-399 元 / PR`

### 4. GitHub Actions 基础落地

交付内容：

- 基础 workflow
- Node / Python 最小 CI
- GitHub Actions 安全基线说明
- 调试建议
- 后续升级方向

建议价格：

- `1999-4999 元 / 仓库`

## 当前最小执行层

这套仓库不再只是资料库，也补了两份最小可执行脚本：

- `automation/ci_red_triage/new-ci-red-report.ps1`
- `automation/pr_review/new-pr-review-report.ps1`

它们当前的作用不是自动替代人工判断，而是把输出骨架固定下来，减少交付格式漂移。

## 交付链条

标准链条是：

1. 判断资料入口是否安全
2. 收集客户信息
3. 确认访问权限与责任边界
4. 执行体检、诊断或初审
5. 用 AI 做辅助分析
6. 人工复核
7. 输出标准交付包
8. 跟进升级到落地包或月度托管

## 安全与边界

默认不做：

- 账号共享
- 代登录
- 海外模型访问转售
- 无边界的全自动修复
- 生产部署托管
- 未脱敏敏感资料处理

必须人工确认的内容：

- 登录
- 支付
- 权限控制
- 数据库迁移
- secrets
- 生产配置
- 任何可能改动业务逻辑的修复

公开仓库不是客户私有资料的默认入口。
如果涉及私有仓库、私有 PR、原始日志、截图或 workflow 文件，必须先走私下渠道，并使用权限授权模板。

## GitHub Actions 最小策略

当前默认策略是：

- 用最小 `permissions`
- 默认只在 `pull_request` 和 `push` 上跑基础检查
- 默认不启用自动 artifact 上传
- 只有在仓库负责人明确确认后，才启用低风险 debug artifact 上传

Node 模板的实际逻辑：

- `lint` 存在就跑
- `test` 存在就跑
- `build` 存在就跑

Python 模板的实际逻辑：

- 默认安装依赖
- 默认确保 `pytest` 可运行
- 默认执行 `pytest`
- 不默认假设存在统一的 `lint` 或 `build` 命令

## 标准成交路径

最稳的路径是：

1. 先卖仓库体检
2. 再卖 GitHub Actions 的 CI 红灯诊断
3. 再卖 GitHub Actions 基础落地
4. 最后转月度 PR/CI 托管

## 关键指标

试点期先看这些指标：

- 首次回复时长
- 平均交付时长
- 根因判断准确率
- 红灯转绿比例
- 试点转付费率
- 客户复购率

## 能力升级路线

### 阶段一：服务型 MVP

目标：

- 用固定模板交付结果
- 用低风险入口包验证需求

### 阶段二：半自动执行层

目标：

- 把高重复动作变成固定脚本
- 降低手工整理成本

### 阶段三：GitHub 原生自动化

目标：

- 在 GitHub 内完成更多结构化动作
- 从一次性服务向产品化能力推进

### 阶段四：托管与扩展

目标：

- 从单次合作走向月度服务
- 增加分层 CI 能力、PR 自动化与更强安全基线

## 当前最现实的业务目标

不是继续堆功能，而是先做好三件事：

1. 拿下第一批真实试点
2. 用低风险入口包验证客户是否愿意付费
3. 用交付质量和边界清晰度建立复购基础
