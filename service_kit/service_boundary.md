# Day 1 - PR/CI 服务边界

## 今日目标

把第一阶段服务定义清楚：我们不是做完整 SaaS，也不是承诺全自动改代码，而是先做一个能收费、能交付、能验证需求的服务型 MVP。

第一阶段服务名称：

**GitHub 仓库体检 + CI 红灯诊断 + PR/CI 自动化落地包**

## 我们提供什么

1. 仓库体检

检查客户 GitHub 仓库的基础工程化状态，包括 README、依赖安装说明、测试命令、CI 配置、PR 模板、secrets 风险、分支保护建议。

2. CI 红灯诊断

客户提供失败的 CI 链接或日志后，我们判断失败原因，提取关键错误，给出修复建议和验证步骤。

3. 基础 GitHub Actions 落地

帮客户配置最基础的 GitHub Actions workflow，让仓库能自动跑 test、lint、build 中的一项或多项。

4. PR 初审辅助

对一个 PR 的 diff 做结构化审查，指出明显 bug、缺失测试、风险改动和需要人工确认的地方。

5. 交付报告

每次服务都输出一份清晰报告，包含问题摘要、关键证据、建议动作、风险等级和下一步。

## 我们暂时不提供什么

1. 不承诺全自动修复所有 CI。

2. 不承诺自动修改业务逻辑代码。

3. 不接生产环境部署责任。

4. 不接客户账号共享、代登录、代注册。

5. 不把海外模型账号转售给中国用户。

6. 不处理明显包含敏感个人信息、生产密钥、重要数据的日志，除非客户先完成脱敏。

7. 不做面向公众开放的 AI 入口站。

## 第一阶段允许 AI 做什么

1. 帮忙总结 CI 日志。

2. 帮忙定位失败原因。

3. 帮忙生成修复建议。

4. 帮忙审查 PR diff。

5. 帮忙生成报告初稿。

## 第一阶段必须人工复核什么

1. AI 对失败原因的判断是否准确。

2. AI 的修复建议是否会误导客户。

3. 是否包含敏感信息、token、密钥、生产地址。

4. 是否涉及业务逻辑、权限、支付、登录、安全等高风险代码。

5. 是否需要客户确认后才能继续。

## 风险分级

低风险：

- lint 错误
- format 错误
- 依赖安装失败
- 测试命令缺失
- README 不清楚
- workflow 配置缺基础步骤

中风险：

- 测试失败但原因明确
- 构建失败但只影响开发环境
- 依赖版本冲突
- 前端页面冒烟失败

高风险：

- 登录、支付、权限、数据删除相关代码
- 数据库迁移
- 生产部署
- secrets、token、私钥、生产配置
- 客户真实用户数据或敏感日志

高风险内容只给分析建议，不自动修改。

## Day 1 完成标准

完成以下三份材料：

1. 服务边界说明

2. 一页服务介绍

3. 客户信息收集表

这三份材料完成后，就可以进入 Day 2：准备样板仓库和 demo 案例。
