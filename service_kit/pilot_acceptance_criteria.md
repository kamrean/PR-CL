# 试点验收标准

## 目标

试点不能只靠“客户觉得还行”来判断是否成功。需要有明确验收标准。

## 通用验收标准

每个试点至少满足以下 5 点中的 4 点，才算成功：

1. 客户能看懂交付内容
2. 客户知道下一步怎么做
3. 交付没有敏感信息问题
4. 结论基本准确
5. 客户愿意继续沟通或继续合作

## 仓库体检试点验收

成功标准：

- 客户认可最重要的 3 个问题
- 客户认可优先级建议
- 报告清楚说明了 README、CI、测试等问题
- 客户愿意至少继续补其中一项

失败信号：

- 客户觉得问题太泛
- 客户看不懂为什么这些问题重要
- 交付不能转成下一步动作

## CI 红灯诊断试点验收

成功标准：

- 能明确失败步骤
- 能提取关键日志
- 根因判断基本正确
- 修复建议至少有一条可执行
- 客户认为这份诊断比自己看日志省时间

失败信号：

- 只给出模糊建议
- 没有验证命令
- 客户看完还是不知道先做什么

## PR 初审试点验收

成功标准：

- 改动摘要准确
- 问题指出得具体
- 风险等级判断合理
- 客户或 reviewer 认为这轮初审有帮助

失败信号：

- 只有空泛建议
- 没指出具体文件或逻辑
- 客户觉得和自己看没区别

## GitHub Actions 落地试点验收

成功标准：

- workflow 能自动触发
- 至少一项检查能稳定运行
- 失败时能看出失败步骤
- 客户理解后续如何自己继续维护

失败信号：

- workflow 不能稳定跑
- 客户不知道哪里改了
- 文档不清楚

## 商业验收标准

如果满足以下任意两项，说明试点有商业价值：

1. 客户愿意继续下一步合作
2. 客户愿意付费购买下一项服务
3. 客户愿意做匿名案例
4. 客户愿意介绍别人

## 试点结束后的判断

试点结束后，你必须明确把每个客户归类成：

- 值得继续推进
- 适合转付费
- 可以留作案例
- 暂不跟进
