# 案例展示页

## 为什么要做案例展示

客户买这种服务，最担心的不是“你会不会说”，而是：

1. 你有没有真的处理过类似问题
2. 你交付出来的东西长什么样
3. 你到底能不能把问题讲清楚

案例展示的目标不是吹得很大，而是让客户快速看到：

- 问题是什么
- 你做了什么
- 结果是什么
- 为什么值得继续合作

## 案例 1：前端 CI 红灯诊断

### 客户场景

一个前端项目在 GitHub Actions 里构建失败，错误日志里出现：

```text
Module not found: Can't resolve './components/Button'
```

### 我们做了什么

1. 定位失败步骤是 `npm run build`
2. 提取关键日志
3. 判断最可能根因是 import 路径错误、文件大小写不一致或文件未提交
4. 给出验证命令和下一步建议
5. 同时对相关 PR 做初审，判断不建议直接合并

### 交付结果

客户能拿到：

- 一句话结论
- 失败步骤
- 关键日志
- 根因判断
- 修复建议
- 验证命令
- 风险等级

### 这个案例值钱的地方

不是“帮客户看了一眼日志”，而是把一条红灯变成客户能执行的下一步。

## 案例 2：旧仓库工程化体检

### 客户场景

一个旧仓库能跑，但缺少 README 说明、没有 CI、没有 PR 模板，也没有明确的测试命令。

### 我们做了什么

1. 按固定 7 项做仓库体检
2. 给出总分和健康等级
3. 指出最重要的 3 个问题
4. 给出最值得先补的 3 个动作
5. 推荐下一步做 GitHub Actions 落地

### 交付结果

客户能拿到：

- 总分
- 健康等级
- 固定检查项结果
- 风险等级
- 修复优先级

### 这个案例值钱的地方

不是“提出一堆建议”，而是帮客户先判断：现在最值得补的不是所有东西，而是基础 CI 和协作保护。

## 案例 3：PR 初审

### 客户场景

一个 PR 试图修复前端构建失败，但 CI 仍未通过，团队不确定是否该合并。

### 我们做了什么

1. 总结 PR 改动摘要
2. 检查潜在 bug
3. 指出缺失测试
4. 判断兼容性风险
5. 给出“暂缓合并”的结论

### 交付结果

客户能拿到：

- 改动摘要
- 问题列表
- 测试建议
- 风险等级
- 是否建议合并

### 这个案例值钱的地方

不是代替正式 reviewer，而是先帮团队把明显风险筛掉，减少 reviewer 的时间消耗。

## 案例展示原则

展示时要遵守：

1. 只展示匿名或已授权案例
2. 不展示敏感日志
3. 不展示 token、key、cookie、私钥
4. 不展示客户不允许公开的截图
5. 每个案例都要有问题、动作、结果三段结构

## 最适合放到首页的案例句式

推荐写法：

- 一个前端仓库 build 红灯，我们把失败步骤、根因判断和修复建议整理成可执行交付包。
- 一个旧仓库工程化很弱，我们用固定 7 项体检把问题排优先级，并建议先补基础 CI。
- 一个 PR 看起来像是修复，但验证不足，我们做了初审后建议暂缓合并。

避免写法：

- 帮客户做了很多事
- 成功提升效率
- 优化了工程流程
