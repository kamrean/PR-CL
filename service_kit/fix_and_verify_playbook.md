# 修复与验证闭环手册

## 目标

把“给建议”推进到“修复、复跑、验收”。

## 闭环步骤

1. 明确问题
2. 提出修复方案
3. 确认是否允许改动
4. 落地修改
5. 复跑验证
6. 记录结果
7. 决定是否升级处理

## 修复前要确认

- [ ] 这次是只做报告，还是允许改动
- [ ] 是否允许建 PR
- [ ] 是否允许改 workflow
- [ ] 是否涉及高风险模块

## 复跑验证要留下什么

- 执行了什么命令
- 跑在什么环境
- 成功还是失败
- 失败时卡在哪一步
- 是否比之前更进一步

## 成功标准

至少满足其一：

- CI 从红变绿
- 失败步骤被明确缩小
- 客户可复现且可继续处理

## 升级条件

遇到以下情况就不要继续自动推进：

- 涉及 secrets
- 涉及权限或生产配置
- 涉及数据库迁移
- 修复会明显改业务逻辑

## 交付里要写清楚

- 这次改了什么
- 为什么这样改
- 怎么验证
- 还有哪些风险没有动
