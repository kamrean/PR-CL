# 试点转付费 SOP

## 目标

试点不是终点，真正的目标是把“你先帮我看看”转成“我愿意继续付费合作”。

## 转付费最好的时机

不是在试点开始前，也不是在试点还没交付时。

最好的时机是：

1. 客户已经看到结果
2. 客户承认结果对他有帮助
3. 你已经指出下一步更大的价值

## 转付费的基本逻辑

### 情况 1：客户买了仓库体检

推荐升级：

- GitHub Actions 落地
- CI 红灯诊断

推荐话术：

```text
这次体检已经把最值得先补的点找出来了。如果你愿意继续，我建议下一步直接把最基础的 GitHub Actions 补起来，这样以后每次 PR 都能自动检查，问题不会一直靠人工发现。
```

### 情况 2：客户买了 CI 红灯诊断

推荐升级：

- 修复 PR
- GitHub Actions 优化
- 月度 CI 托管

推荐话术：

```text
这次红灯已经定位清楚了。你如果不想以后每次都临时救火，可以继续做两种升级：一种是我帮你把这次修复整理成标准化做法，另一种是直接做月度 CI 托管。
```

### 情况 3：客户买了 PR 初审

推荐升级：

- 月度 PR 初审托管
- 仓库体检

推荐话术：

```text
如果你们 PR 数量比较稳定，这种初审更适合做成固定服务。这样每次提交都能先过一轮风险筛查，正式 reviewer 压力会小很多。
```

### 情况 4：客户买了 GitHub Actions 落地

推荐升级：

- 月度 PR/CI 托管
- CI 红灯自动诊断

推荐话术：

```text
基础 CI 已经搭好了，下一步最自然的升级就是让红灯出来时更快定位，或者把 PR 初审和 CI 诊断做成固定托管。
```

## 不要怎么转付费

避免：

- 刚交付完就立刻催下单
- 一次性推所有服务
- 没有结合客户这次真实问题
- 用很空的“长期合作”这种话术

## 推荐转付费提问

1. 这次交付里，哪部分对你最有价值？
2. 你更想先解决仓库基础问题，还是减少以后看红灯的时间？
3. 如果继续合作，你更倾向单次还是包月？

## 最终目标

试点后至少推动客户进入下面一个状态：

- 继续下一项单次服务
- 转月度托管
- 愿意做案例和转介绍
- 未来有明确再次合作窗口
