# Day 3 - 仓库体检风险分级标准

## 使用目的

这份标准用于把仓库体检发现的问题分成高、中、低三个等级，方便客户决定先修什么。

## 高风险

定义：

可能导致安全事故、生产事故、主分支污染、无法交付的问题。

典型问题：

- 代码中出现 API key、token、private key
- .env 被提交到仓库
- CI 日志打印 secrets
- main/master 分支可以直接 push
- 没有任何 CI，但团队已经多人协作
- CI 完全跑不起来
- 测试命令不存在，且没有任何验证方式
- workflow 权限过大
- 涉及登录、支付、权限、数据删除的改动没有 review 机制

处理建议：

- 立即处理
- 不建议自动修复业务代码
- 需要客户确认权限和安全边界
- 可能需要先轮换已泄露密钥

客户解释：

这类问题会直接影响安全、交付和团队协作，应该优先修。

## 中风险

定义：

短期不一定造成事故，但会明显增加协作成本、CI 红灯概率和交付不稳定性。

典型问题：

- README 有介绍但缺少运行步骤
- 依赖文件存在但没有版本说明
- 没有 lock 文件
- 有测试但 README 没说明怎么跑
- CI 只在 push 时跑，不在 PR 时跑
- PR 模板过于简单
- 没有明确的环境变量示例
- build/test/lint 只覆盖一部分关键流程
- CI 失败日志很难读

处理建议：

- 排进近期修复计划
- 可以由我们提供模板化修复
- 大多数属于低到中等技术风险

客户解释：

这类问题会让项目越来越难维护，尤其在多人协作、交付外包、接 AI 自动化时会放大。

## 低风险

定义：

不影响当前运行，但会影响专业度、可读性和后续扩展。

典型问题：

- README 排版一般
- 缺少贡献说明
- 缺少 issue 模板
- 缺少截图或示例
- workflow 名称不清楚
- PR 模板缺少少量字段
- 没有 badge
- 没有常见问题说明

处理建议：

- 可以在主要问题修完后优化
- 适合用模板批量补齐
- 可作为仓库现代化套餐的一部分

客户解释：

这类问题不一定马上出事故，但会影响仓库给外部协作者、客户或 AI 工具的可读性。

## 优先级判断公式

优先级 = 风险等级 + 出现频率 + 修复成本

优先处理：

1. 高风险且修复成本低的问题
2. 会阻塞 CI 或协作的问题
3. 会泄露 secrets 或污染主分支的问题
4. 客户当前正在痛的问题

可以后置：

1. 只影响美观的问题
2. 暂时没有多人协作需求的问题
3. 需要客户内部决策的问题

## 对外沟通原则

1. 不吓唬客户，但要明确风险。

2. 每个问题都给可执行建议。

3. 不只说“有问题”，要说“为什么影响你”。

4. 高风险问题先要求客户确认，再继续深度处理。

5. 如果发现疑似密钥泄露，优先建议客户轮换密钥。
