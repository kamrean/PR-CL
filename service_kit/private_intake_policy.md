# 私有资料接单政策

## 目标

这份政策用来明确一件事：

`公开仓库不是客户敏感资料的默认入口。`

对于这套 `仓库体检 / CI 红灯诊断 / PR 初审 / GitHub Actions 落地` 服务，任何可能包含客户内部信息的内容，都必须先判断是否适合公开提交。

## 公开仓库里可以提交什么

只允许提交下面这类信息：

- 公开仓库链接
- 已公开的 PR 链接
- 已完全脱敏的极短错误摘要
- 不包含客户身份和内部路径的功能需求
- 公开 demo 仓库相关问题

## 公开仓库里禁止提交什么

禁止在公开 issue、公开 PR、公开评论里提交以下内容：

- 私有仓库链接
- 私有 PR 链接
- Commit SHA
- 原始 CI 长日志
- 截图里的密钥、内网地址、数据库地址
- token、key、cookie、private key
- 客户真实姓名、公司名、邮箱、手机号
- 生产环境配置
- 任何未确认完成脱敏的材料

## 私有资料应该怎么收

如果需求涉及以下任一项，就必须转私下渠道收集：

- 私有仓库
- 私有 PR
- 私有 CI 日志
- 测试截图
- GitHub Actions 失败截图
- workflow 文件
- 权限申请
- 任何需要查看真实代码或真实配置的场景

建议使用的私下渠道：

- 私聊
- 邮件
- 客户指定企业沟通工具
- 临时共享文档
- 只读仓库访问

## 公开 issue 的正确用途

公开 issue 只适合两类场景：

1. 公开 demo 仓库的演示问题
2. 不包含敏感信息的合作意向登记

如果客户要提交真实项目资料，公开 issue 只能作为“先联系我”的入口，不能作为正式资料收集表。

## 接单规则

- 对私有项目，默认先发这份政策给客户
- 对公开项目，也要先提醒不要粘贴长日志
- 一旦看到疑似敏感信息，立即停止继续收集
- 要求客户先撤回、删除或脱敏后再继续

## 一句话规则

`公开入口只收公开信息，真实资料走私下渠道。`
