# 试点执行包

## 目标

这份执行包用于第一次真实试点合作，确保你不是“边做边想”，而是按固定流程推进。

## 适用范围

适合以下 4 类试点：

1. 仓库体检
2. CI 红灯诊断
3. PR 初审
4. GitHub Actions 落地

## 试点开始前必须确认

1. 客户是谁
2. 当前最痛的问题是什么
3. 仓库或日志是否已脱敏
4. 本次试点只做哪一个入口服务
5. 交付时间是什么时候
6. 是否允许匿名沉淀案例

## 试点执行流程

### Step 1：确认范围

每次试点只能有一个核心目标。

例如：

- 只做一次仓库体检
- 只做一条 CI 红灯诊断
- 只做一个 PR 初审
- 只做一个基础 workflow 落地

不要在第一单里同时答应：

- 体检
- 修 CI
- 看 PR
- 配 workflow
- 顺手改业务代码

### Step 2：收集输入

根据不同服务，至少收这些材料：

仓库体检：

- 仓库链接
- 技术栈
- 是否公开

CI 红灯诊断：

- CI 链接
- 关键日志
- 失败命令

PR 初审：

- PR 链接
- 改动背景
- CI 状态

GitHub Actions 落地：

- 仓库链接
- 安装命令
- 测试命令
- lint 命令
- build 命令

### Step 3：先做低风险判断

必须先判断：

- 是否涉及生产
- 是否涉及 secrets
- 是否涉及登录/支付/权限
- 是否涉及数据库
- 是否有真实用户数据

如果是高风险，先降级成“只出诊断和建议”，不要直接承诺修改。

### Step 4：按模板执行

只使用仓库里已有模板：

- `repo_audit_*`
- `ci_red_*`
- `pr_review_*`
- `delivery_pack_*`

不要一单一套新格式。

### Step 5：交付前复核

必须复核：

1. 是否有敏感信息
2. 是否有编造内容
3. 是否有可执行建议
4. 是否有下一步动作
5. 是否需要客户确认

### Step 6：交付

每次试点统一给客户：

1. 一句话结论
2. 最重要的 3 个问题
3. 下一步建议
4. 详细报告
5. 风险确认

### Step 7：收反馈

交付完成后 24 小时内收第一轮反馈。

必须问：

1. 最有帮助的地方是什么
2. 最不清楚的地方是什么
3. 你愿不愿意继续下一步服务

## 第一单最重要的原则

1. 范围要小
2. 结论要清楚
3. 风险要明确
4. 不要过度承诺
5. 一定要留下可复用案例
