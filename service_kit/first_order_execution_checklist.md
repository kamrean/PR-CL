# 首单执行清单

## 目标

这份清单用于你拿下第一单时，避免出现下面几种常见问题：

- 还没确认边界就开始接活
- 客户只发一句“帮我看看 CI 为什么红了”
- 报价之前没判断风险
- 做完了，但交付物不够专业
- 交付后没有顺手推进下一单

这份清单默认围绕 3 类服务展开：

1. 仓库体检
2. CI 红灯诊断
3. PR/CI 自动化落地包

## 一、接单前

- [ ] 判断客户属于哪一类：学生项目、独立开发者、小外包团队、小 SaaS 团队、旧仓库维护者
- [ ] 判断客户现在最急的问题是：仓库混乱、CI 红灯、PR review 慢、没有基础 workflow
- [ ] 明确这次只卖一个入口包，不同时卖太多内容
- [ ] 默认优先卖低风险入口包

推荐入口顺序：

1. 仓库体检
2. CI 红灯单次诊断
3. GitHub Actions 落地
4. 月度托管

## 二、拿资料前

- [ ] 让客户发仓库链接
- [ ] 如果是私有仓库，先确认是否能给只读访问
- [ ] 如果不能给访问，就让客户发脱敏后的截图、日志、目录结构
- [ ] 让客户说明项目语言和框架
- [ ] 让客户说明这次最想解决的一件事

最低必收资料：

- 仓库链接或仓库截图
- 当前失败日志或报错截图
- 运行命令
- 预期结果
- 当前卡住的位置

## 三、风险判断

在报价前先判断有没有超出低风险边界。

低风险，可先接：

- README 缺失或混乱
- 安装说明不清楚
- lint 失败
- test 失败
- build 失败
- GitHub Actions 没配好
- PR 模板缺失

中高风险，先缩范围再接：

- 登录
- 支付
- 权限控制
- 数据库迁移
- secrets
- 生产部署
- 客户真实数据

## 四、报价前确认

- [ ] 确认这次交付是单次服务还是小包交付
- [ ] 确认是否只做诊断，不负责修改代码
- [ ] 确认是否需要你落地 workflow
- [ ] 确认交付时间
- [ ] 确认客户能接受“先低风险、再升级”

报价前必须说清楚：

- 这次卖的是结果，不是长期兜底
- 这次是单仓库还是多仓库
- 是否包含修改文件
- 是否包含复测
- 是否包含二次沟通

## 五、开始执行

### 如果卖的是仓库体检

- [ ] 跑固定检查项
- [ ] 记录 3 个最高优先级问题
- [ ] 给出 3 个先做动作
- [ ] 标记低、中、高风险
- [ ] 输出正式体检报告

### 如果卖的是 CI 红灯诊断

- [ ] 提取失败步骤
- [ ] 提取关键日志
- [ ] 判断最可能根因
- [ ] 给出修复建议
- [ ] 给出验证命令
- [ ] 标记是否适合自动修复

### 如果卖的是 GitHub Actions 落地

- [ ] 判断项目类型：Node、Python、混合项目
- [ ] 复制最小 workflow 模板
- [ ] 补齐 install、lint、test、build
- [ ] 配置失败时上传 artifact
- [ ] 先本地检查，再交付客户
- [ ] 给出后续优化建议

## 六、交付前检查

- [ ] 没有泄露 token、key、cookie、账号信息
- [ ] 所有日志都做了脱敏
- [ ] 报告里没有空话
- [ ] 每条建议都能落地执行
- [ ] 风险边界写清楚
- [ ] 下一步建议写清楚

## 七、正式交付

交付包至少要包含：

- 一句话结论
- 最重要的 3 个问题
- 建议优先做的 3 个动作
- 正式报告
- 风险说明
- 下一步建议

## 八、交付后跟进

- [ ] 24 小时内确认客户是否看完
- [ ] 问客户最有价值的是哪一条建议
- [ ] 问客户是否愿意继续做下一步
- [ ] 如果客户满意，申请案例授权
- [ ] 如果客户没有继续，记录原因

## 九、首单最重要的目标

第一单不要追求利润最大化，先追求这 4 件事：

1. 交付物专业
2. 边界清楚
3. 客户看得懂
4. 能顺着升级到下一单

## 十、首单推荐成交路径

最稳的路径是：

1. 先卖一次仓库体检或 CI 红灯诊断
2. 用交付报告建立信任
3. 再卖 GitHub Actions 落地
4. 最后再谈月度托管
