# 客户筛选与优先级评分

## 目标

不是所有客户都适合做第一批。前 10 个客户最重要的是：

1. 容易成交
2. 风险低
3. 有案例价值
4. 反馈清楚

## 评分维度

每项 0-2 分，总分 10 分。

### 1. 痛点清晰度

- 0 分：只说“想看看能做什么”
- 1 分：知道有问题，但说不清
- 2 分：明确说出是 README、CI、PR 或 GitHub Actions 问题

### 2. 技术风险

- 0 分：涉及生产、数据库、支付、权限、敏感信息
- 1 分：中等复杂度，有私有仓库或配置依赖
- 2 分：公开仓库或低风险问题

### 3. 决策速度

- 0 分：需要复杂审批
- 1 分：需要多人确认
- 2 分：个人或小团队负责人可以直接决定

### 4. 案例价值

- 0 分：不允许展示，也没有代表性
- 1 分：有一定代表性，但不能公开
- 2 分：问题典型、结果容易展示

### 5. 升级可能

- 0 分：只想问一个零散问题
- 1 分：有继续合作可能
- 2 分：明显适合从体检升级到 CI 或托管

## 结果判断

- 8-10 分：优先联系
- 6-7 分：可以试点
- 4-5 分：谨慎接
- 0-3 分：不建议做前期客户

## 最适合的前期客户

1. 有公开 GitHub 仓库的独立开发者
2. 学生项目队
3. 小外包团队
4. 有真实红灯但问题不涉高风险系统的客户

## 不建议前期接的客户

1. 一上来就要全自动修生产问题
2. 要求代登录或账号共享
3. 提供未脱敏敏感日志
4. 没明确问题，但想要“整体咨询”
5. 高沟通成本、低付费意愿
