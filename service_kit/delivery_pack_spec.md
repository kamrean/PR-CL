# Day 8 - 标准交付包规范

## 今日目标

把前面已经做好的服务内容，整理成客户真正能验收的“交付包”。

交付包的作用不是堆文件，而是让客户一眼知道：

1. 发现了什么问题
2. 为什么重要
3. 下一步怎么做
4. 哪些风险需要确认

## 适用服务

这份交付包规范适用于：

1. 仓库体检
2. CI 红灯诊断
3. PR 初审
4. GitHub Actions 落地

## 标准交付包结构

建议每次交付都整理成下面这个结构：

```text
delivery-pack/
  00-summary.md
  01-detailed-report.md
  02-next-steps.md
  03-risk-and-confirmation.md
  artifacts/
  screenshots/
```

## 每个文件的作用

### 00-summary.md

客户最先看这一份。

必须回答：

1. 这次主要发现了什么
2. 最重要的 3 个问题是什么
3. 建议先做哪 3 个动作
4. 风险等级是什么

### 01-detailed-report.md

这是详细报告。

内容来源：

- 仓库体检报告
- CI 红灯诊断报告
- PR 初审报告

要求：

- 有证据
- 有结论
- 有建议

### 02-next-steps.md

这是客户最容易真正执行的一份。

必须写成动作清单：

1. 今天先做什么
2. 本周再做什么
3. 后续可以升级什么

### 03-risk-and-confirmation.md

把高风险和需要客户拍板的事项单独列出来。

例如：

- 是否允许修改依赖版本
- 是否允许修改 workflow
- 是否允许创建修复 PR
- 是否涉及 secrets
- 是否涉及生产配置

### artifacts/

这里放机器可读材料：

- 关键日志
- workflow 文件
- diff
- test 结果
- 失败截图

### screenshots/

这里放客户容易理解的材料：

- CI 红灯截图
- GitHub Actions 截图
- PR 页面截图
- 前后对比图

## 包名命名规则

建议：

```text
客户名_服务类型_日期
```

例如：

```text
demo-repo_ci-red_2026-04-24
kamrean_repo-audit_2026-04-24
student-team_pr-review_2026-04-24
```

## 交付包最小验收标准

一份合格交付包必须满足：

1. 客户不看原始日志也能知道问题。

2. 客户能知道下一步怎么做。

3. 高风险事项单独列出。

4. 不包含未脱敏敏感信息。

5. 不只是“分析”，而是有可执行建议。
