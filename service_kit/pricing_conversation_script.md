# 实际报价话术

## 目标

这份话术不是让你“背销售台词”，而是帮你在报价时做到三件事：

1. 不把服务说虚
2. 不一上来报大单
3. 让客户自然接受先做低风险入口包

## 一、报价原则

先判断，再报价。

你要先判断：

- 客户到底卡在哪里
- 这次是诊断类需求，还是落地类需求
- 风险有没有超出边界
- 这次是单次合作，还是有机会延伸到后续合作

## 二、最常用的开场说法

### 场景 1：客户只说“帮我看看”

推荐说法：

“可以，我建议这次先从一个低风险入口包开始，不先做大而全。你先把仓库链接或脱敏后的 CI 日志发我，我先判断更适合做仓库体检还是单次 CI 红灯诊断。”

### 场景 2：客户说“能不能直接帮我全搞定”

推荐说法：

“可以往那个方向走，但第一步不建议直接做全自动或全托管。更稳妥的是先做一次体检或诊断，把问题和风险边界看清楚，再决定要不要继续做落地和托管。”

### 场景 3：客户说“预算不多”

推荐说法：

“那更适合先做入口包。比如一次仓库体检，或者一次 CI 红灯诊断。这样你能先看到报告质量和建议是否有用，再决定要不要继续。”

## 三、推荐报价顺序

### 1. 仓库体检

适合这样说：

“如果你现在还不确定仓库到底乱在哪，最适合先做仓库体检。我会按固定项检查 README、依赖安装、测试命令、CI 配置、secrets 风险、PR 模板和分支保护，最后给你一份正式报告。这个包一般在 499 到 999 元一个仓库。”

### 2. CI 红灯单次诊断

适合这样说：

“如果你现在最急的是 CI 红灯，我建议先做单次诊断。我会提取关键日志、判断最可能根因、给你修复建议和验证命令。这种适合作为快入口，一般 99 到 299 元一次。”

### 3. PR 初审

适合这样说：

“如果你主要担心 PR 合进去出问题，可以先做一次 PR 初审。我会把潜在 bug、缺失测试、兼容性和高风险修改点给你标出来。这种通常 99 到 399 元一个 PR。”

### 4. GitHub Actions 落地

适合这样说：

“如果你已经确定要把流程补起来，我可以帮你落地一套基础 GitHub Actions，包括最小 CI、失败 artifact、调试建议和后续优化建议。这个通常按仓库报价，1999 到 4999 元。”

### 5. 月度托管

适合这样说：

“如果你每个月都会遇到 PR review、CI 红灯和工程化维护问题，那更适合做月度托管，不按单次反复买。通常 999 到 4999 元每月，看仓库数量和频率。”

## 四、报价时一定要带上的边界

报价时建议顺手补这句话：

“这次报价默认不包含登录、支付、权限、数据库迁移、生产部署和客户真实数据处理；如果涉及这些内容，我会先单独缩范围，再重新确认报价。”

## 五、客户常见压价时怎么说

### 客户说：“能不能便宜点？”

推荐说法：

“可以，我们不一定降质量，可以先缩范围。比如这次只做单次诊断，不包含落地；或者只看一个仓库、不看多仓库。这样更稳，也更适合第一次合作。”

### 客户说：“别人更便宜。”

推荐说法：

“可以理解。我的做法不是只给几句口头建议，而是给可交付的报告、明确的修复方向和下一步动作。你第一次也可以先用最小入口包试一下，判断是否值这个价。”

### 客户说：“你先做完我看看再说。”

推荐说法：

“第一次合作可以给低价入口包，但不建议做无边界试做。更稳的是先约一个最小交付，比如一次体检或一次诊断，这样双方边界都清楚。”

## 六、客户同意后的确认句

客户准备成交时，发这段最稳：

“我再确认一下这次合作范围：本次只做 `仓库体检 / CI 红灯诊断 / GitHub Actions 落地` 中的一个包；交付时间是 `X`；交付内容包含 `报告 / 建议 / workflow / 验证命令`；不包含高风险业务改动和生产环境操作。你确认后我就开始。”

## 七、最适合首单的组合

首单最推荐这样卖：

1. 先卖仓库体检或 CI 红灯单次诊断
2. 交付报告后，再追加 GitHub Actions 落地
3. 客户满意后，再谈月度托管

## 八、不要这样报价

避免这些说法：

- “什么都能修”
- “我帮你全自动搞定”
- “肯定一次解决”
- “你把权限都给我就行”
- “先不管边界，做完再说”

这些说法虽然容易让客户点头，但后面最容易把自己拖进高风险交付。
