# 试点邀约文案

## 目标

试点的目的不是“免费劳动”，而是用低风险、小范围合作换：

1. 真反馈
2. 真案例
3. 真成交路径

## 试点原则

1. 试点必须限定范围。

2. 试点必须限定交付物。

3. 试点必须换反馈或案例授权。

4. 试点不能做成无上限支持。

## 试点方案 1：免费简版体检

适合：

- 熟人项目
- 公开仓库
- 想快速做样例

文案：

```text
我现在在做一套 GitHub 工程化服务，想找 1-2 个公开仓库做试点样例。

我可以先免费做一个“简版仓库体检”，内容包括：

1. 最重要的 3 个问题
2. 最建议先补的 3 个动作
3. 是否适合接 GitHub Actions 和 PR 自动检查

我希望换到的是：

1. 你对报告质量的真实反馈
2. 如果你觉得有用，允许我匿名把这个案例写进我的 demo
```

## 试点方案 2：低价 CI 红灯诊断

适合：

- 真实红灯
- 客户不想一次投入太大

文案：

```text
如果你现在有一条真实 CI 红灯，我可以先按低价试点做一次完整诊断。

你会拿到：

1. 失败步骤
2. 关键日志
3. 根因判断
4. 修复建议
5. 验证命令

试点阶段的目标不是长期托管，而是先验证这项服务对你到底有没有价值。
```

## 试点方案 3：低价 PR 初审样例

适合：

- 小团队
- 独立开发者
- 没有固定 reviewer

文案：

```text
如果你最近正好有一个 PR 想先过一轮风险筛查，我可以按试点价先做一个 PR 初审样例。

交付会包括：

1. 改动摘要
2. 潜在 bug
3. 缺失测试
4. 风险等级
5. 是否建议合并

如果这轮合作有帮助，后面我们再考虑做月度 PR/CI 托管。
```

## 试点方案 4：低价 GitHub Actions 落地

适合：

- 没有 CI 的仓库
- 学生项目
- 小团队内部工具

文案：

```text
如果你仓库目前还没有 GitHub Actions，我可以先按试点价帮你做一个最小版本。

第一版只做：

1. install
2. test / lint / build 中最基础的一项或几项
3. 失败日志可读

这样你可以先验证“自动检查”到底能不能帮你减少问题，而不是一开始就上复杂自动化。
```

## 试点结束后要问的 5 个问题

1. 这份交付最有用的地方是什么？
2. 哪部分最不清楚？
3. 你愿不愿意为这个服务付费？
4. 你更愿意买单次还是包月？
5. 你愿不愿意让我匿名展示这个案例？
