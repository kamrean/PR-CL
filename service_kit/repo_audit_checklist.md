# Day 3 - 仓库体检检查清单

## 使用方式

这份清单用于检查一个 GitHub 仓库是否具备基本工程化能力。每一项都按 0-2 分打分。

评分说明：

- 0 分：缺失或明显不可用
- 1 分：有但不完整
- 2 分：清晰、可用、适合团队协作

总分 14 分。

结果判断：

- 12-14 分：仓库基础健康
- 8-11 分：可以协作，但需要补齐关键流程
- 4-7 分：工程化较弱，容易出现交付问题
- 0-3 分：不适合多人协作或自动化交付

## 1. README

检查目标：

判断新人或外部协作者能不能看懂这个项目是什么、怎么运行、怎么测试。

检查项：

- 是否说明项目用途
- 是否说明技术栈
- 是否说明安装依赖步骤
- 是否说明本地启动命令
- 是否说明测试命令
- 是否说明环境变量配置
- 是否有常见问题或故障说明

打分：

- 0 分：README 缺失，或只有项目名
- 1 分：README 有基本介绍，但缺少运行/测试/配置说明
- 2 分：README 能让新人按步骤跑起项目

客户可读解释：

README 是仓库的说明书。如果说明书不清楚，新人接手、外包交付、AI 分析都会变慢。

## 2. 依赖安装

检查目标：

判断项目依赖是否能被稳定安装。

检查项：

- Node 项目是否有 package.json
- Node 项目是否有 lock 文件，如 package-lock.json、pnpm-lock.yaml、yarn.lock
- Python 项目是否有 requirements.txt、pyproject.toml 或 poetry.lock
- 是否说明 Node/Python 版本
- 是否有安装命令
- 是否存在明显过期或冲突依赖

打分：

- 0 分：看不出如何安装依赖
- 1 分：有依赖文件，但缺少版本/安装说明
- 2 分：依赖文件和安装步骤清楚，可复现

客户可读解释：

依赖安装不稳定会导致“我电脑能跑，你电脑跑不了”，也会让 CI 经常红灯。

## 3. 测试命令

检查目标：

判断项目是否有可验证的自动测试入口。

检查项：

- 是否存在测试文件
- 是否有 test 命令
- package.json scripts 是否包含 test
- Python 项目是否能用 pytest / unittest 运行
- README 是否说明如何跑测试
- 测试失败时是否能看到明确错误

打分：

- 0 分：没有测试命令，也看不出如何验证
- 1 分：有测试文件或命令，但说明不完整或不可稳定运行
- 2 分：测试命令明确，能作为 CI 检查项

客户可读解释：

测试命令是判断代码有没有被改坏的基础。如果没有测试，AI 或人类都很难确认修复是否真的安全。

## 4. CI 配置

检查目标：

判断仓库是否有自动化检查流程。

检查项：

- 是否存在 .github/workflows 目录
- 是否有 pull_request 或 push 触发
- 是否包含 checkout 步骤
- 是否安装依赖
- 是否运行 test / lint / build
- 是否限制合理权限
- CI 失败日志是否容易读

打分：

- 0 分：没有 CI
- 1 分：有 CI，但只跑了很少步骤或经常不可用
- 2 分：CI 能自动验证关键流程，失败原因可追踪

客户可读解释：

CI 就是代码自动体检。没有 CI，问题通常要到合并后甚至上线前才暴露。

## 5. Secrets 风险

检查目标：

判断仓库是否可能泄露密钥、token、生产配置或敏感信息。

检查项：

- 代码里是否出现 API key / token / private key
- .env 是否被提交
- 是否有 .env.example
- .gitignore 是否忽略本地配置和敏感文件
- GitHub Actions 是否直接打印 secrets
- workflow 是否使用 GitHub Secrets
- 日志里是否可能暴露敏感信息

打分：

- 0 分：发现明显密钥或敏感配置风险
- 1 分：没有明显泄露，但 secrets 管理不规范
- 2 分：有 .env.example、.gitignore、GitHub Secrets 等基本保护

客户可读解释：

Secrets 风险是最高优先级问题。一次密钥泄露可能导致账号被盗、服务被刷、数据被访问。

## 6. PR 模板

检查目标：

判断提交 PR 的人是否知道应该说明什么。

检查项：

- 是否存在 .github/pull_request_template.md
- 是否要求说明改动目的
- 是否要求说明测试方式
- 是否要求勾选风险项
- 是否要求关联 issue 或需求
- 是否提醒截图、日志、兼容性影响

打分：

- 0 分：没有 PR 模板
- 1 分：有模板，但内容太空泛
- 2 分：模板能帮助 reviewer 快速判断风险

客户可读解释：

PR 模板能减少沟通成本，让 reviewer 更快知道这次改动为什么做、怎么验证、风险在哪里。

## 7. 分支保护

检查目标：

判断主分支是否有基本保护，避免未检查代码直接合并。

检查项：

- main/master 是否禁止直接 push
- 是否要求 PR 才能合并
- 是否要求 CI 通过
- 是否要求至少 1 人 review
- 是否限制强制推送
- 是否保护 release 分支

打分：

- 0 分：主分支可随意直接修改
- 1 分：有部分保护，但关键检查不是强制
- 2 分：主分支有 PR、CI、review 等基本保护

客户可读解释：

分支保护是团队协作的安全阀。没有保护时，错误代码很容易直接进入主分支。
