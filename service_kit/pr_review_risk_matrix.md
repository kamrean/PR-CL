# Day 7 - PR 初审风险矩阵

## 使用目的

这份矩阵用于判断一个 PR 初审结果应该给出什么合并建议。

## 总览

| 风险等级 | 常见改动 | 合并建议 |
|---|---|---|
| 低风险 | 文档、模板、低风险 CI 配置 | 可以合并或轻量 review |
| 中风险 | 依赖、测试、构建、普通业务逻辑 | 补充验证后再合并 |
| 高风险 | 登录、支付、权限、数据库、secrets、部署 | 必须人工确认 |

## 低风险 PR

典型改动：

- README 更新
- 文档修正
- PR 模板更新
- Issue 模板更新
- workflow 只新增只读检查
- lint/format 修复

合并建议：

- CI 通过后可以合并
- 如无 CI，建议人工快速确认

注意：

低风险不等于不用看，仍需确认没有误提交敏感信息。

## 中风险 PR

典型改动：

- 修改普通业务逻辑
- 新增测试
- 修改测试断言
- 升级依赖
- 修改构建配置
- 修改 GitHub Actions steps
- 修改环境变量名称

合并建议：

- 暂缓合并，先确认测试和 CI
- 如升级依赖，需要看 changelog 或锁文件变化
- 如修改测试断言，需要确认不是为了掩盖 bug

注意：

中风险 PR 最容易出现“看起来只是小改，但影响流程”的问题。

## 高风险 PR

典型改动：

- 登录逻辑
- 支付逻辑
- 权限判断
- 数据库迁移
- 删除数据
- 生产部署
- secrets/token
- workflow 写权限扩大
- 引入不可信 third-party action

合并建议：

- 需要人工确认
- 不建议只靠 AI 初审通过
- 需要明确 rollback 或验证方案

注意：

高风险 PR 的目标不是快速合并，而是避免事故。

## 合并建议规则

可以合并：

- 低风险
- CI 通过
- 没有明显缺失测试
- 没有 secrets 或权限风险

暂缓合并：

- CI 未通过
- 缺少关键测试
- 修改依赖但没有说明
- 发现潜在 bug
- 改动范围过大但说明不足

需要人工确认：

- 高风险领域
- 信息不足
- AI 判断不确定
- 涉及真实客户数据
- 涉及生产环境

## 对客户的解释方式

不要说：

- 这个 PR 不行。
- AI 觉得有问题。
- 你最好别合。

建议说：

- 当前 PR 涉及权限逻辑，建议由熟悉业务的人再确认一次。
- 当前 CI 未通过，建议先修复红灯后再考虑合并。
- 当前改动缺少回归测试，建议补一个最小测试后合并。
