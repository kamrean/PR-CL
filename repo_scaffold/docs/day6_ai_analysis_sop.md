# Day 6 - AI 辅助 CI 分析 SOP

## 目标

把 CI 日志交给 AI 分析时，避免 AI 乱猜、输出空话或忽略敏感信息。

Day 6 的核心原则：

1. 先人工脱敏，再交给 AI。
2. 只给 AI 必要上下文，不整段乱塞。
3. 固定提示词。
4. 固定输出格式。
5. AI 输出必须人工复核后才能交付客户。

## 输入材料

每次让 AI 分析 CI 红灯前，准备这些内容：

1. 项目语言和框架。
2. CI 系统，例如 GitHub Actions。
3. 失败 workflow / job / step。
4. 失败命令。
5. 关键日志 10-30 行。
6. package.json / requirements.txt / pyproject.toml 中相关片段。
7. 客户期望结果。

不要输入：

- API key
- token
- cookie
- private key
- 生产数据库地址
- 用户个人信息
- 大段无关日志

## 标准提示词

```text
你是 CI 红灯诊断助手。请只根据我提供的信息分析，不要编造不存在的文件或日志。

请按以下格式输出：

1. 失败步骤
2. 关键日志解释
3. 最可能根因
4. 其他可能根因
5. 修复建议
6. 验证命令
7. 风险等级：低/中/高
8. 是否适合自动修复：是/否/需要客户确认
9. 需要人工复核的点

项目背景：
[填写语言、框架、CI 系统]

失败步骤：
[填写 workflow/job/step]

失败命令：
[填写命令]

关键日志：
[粘贴脱敏后的 10-30 行日志]

相关配置：
[粘贴 package.json scripts 或 requirements 等相关片段]
```

## AI 输出验收标准

合格输出必须满足：

1. 明确指出失败步骤。
2. 能解释关键日志。
3. 根因是可验证假设。
4. 修复建议具体到文件、命令或配置。
5. 给出验证命令。
6. 明确风险等级。
7. 不编造客户没有提供的信息。

不合格输出包括：

- “建议检查一下”这种空话。
- 没有验证命令。
- 把猜测写成确定事实。
- 要求客户跳过测试。
- 未提醒高风险权限或 secrets 问题。

## 人工复核清单

交付客户前必须检查：

1. AI 是否误判失败步骤。
2. AI 是否忽略第一条真正错误。
3. 修复建议是否会改业务逻辑。
4. 是否建议删除测试或降低检查强度。
5. 是否涉及 secrets、权限、部署、数据库。
6. 验证命令是否真实存在。
7. 报告里是否残留敏感信息。

## 交付格式

最终交付不要直接贴 AI 原文。必须整理成客户报告：

1. 一句话结论。
2. 关键日志。
3. 根因判断。
4. 修复建议。
5. 验证命令。
6. 风险等级。
7. 是否需要客户确认。

## Day 6 完成标准

完成以下材料：

1. AI 分析标准提示词。
2. AI 输出验收标准。
3. 人工复核清单。
4. 客户交付格式。
