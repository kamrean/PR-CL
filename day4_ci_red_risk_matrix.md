# Day 4 - CI 红灯风险矩阵

## 使用目的

这份矩阵用于判断一次 CI 失败适合怎么处理：直接给建议、尝试自动修复、还是必须客户确认。

## 风险等级总览

| 风险等级 | 适合处理方式 | 是否可自动修复 | 是否需要客户确认 |
|---|---|---|---|
| 低风险 | 直接建议或生成低风险补丁 | 可以 | 通常不需要 |
| 中风险 | 给建议，必要时生成 PR | 谨慎 | 建议确认 |
| 高风险 | 只分析，不自动改 | 不建议 | 必须确认 |

## 低风险 CI 红灯

典型情况：

- lint 失败
- format 失败
- package.json scripts 缺少命令
- workflow 没有安装依赖
- Node/Python 版本明显不匹配
- README 命令和实际命令不一致
- 测试快照需要更新，但业务影响很小

处理方式：

- 可以直接给修复建议
- 可以生成补丁草稿
- 可以建议创建 PR

注意：

即使是低风险，也要复跑验证命令。

## 中风险 CI 红灯

典型情况：

- 单元测试失败
- 构建配置错误
- 依赖版本冲突
- 环境变量缺失
- 类型检查失败
- 前端构建失败
- 第三方服务 mock 缺失

处理方式：

- 先输出诊断报告
- 给 1-3 个修复选项
- 如果要改依赖、测试断言或配置，建议客户确认

注意：

中风险问题常常有多个可能根因，不要把猜测写成确定事实。

## 高风险 CI 红灯

典型情况：

- secrets 泄露
- token 权限不足或权限过大
- 生产部署失败
- 数据库迁移失败
- 支付、登录、权限系统测试失败
- 数据删除、覆盖、迁移相关失败
- CI 使用生产凭证
- workflow 可被外部 PR 滥用

处理方式：

- 只做诊断
- 不自动修复
- 不直接创建改动 PR
- 明确要求客户确认
- 如涉及密钥泄露，建议客户先轮换密钥

注意：

高风险问题不要为了“把红灯变绿”而修改测试或跳过检查。

## 自动修复决策

可以自动修：

- 格式化
- lint
- CI 配置缺基础步骤
- 明确工具版本不匹配
- 文档命令不一致

需要确认再修：

- 升级依赖
- 修改测试断言
- 改构建配置
- 改环境变量
- 改权限配置

禁止自动修：

- secrets
- 生产部署
- 数据库迁移
- 登录/支付/权限核心逻辑
- 删除或重写大量代码

## 客户沟通句式

低风险：

这个问题属于低风险配置/规范问题，可以先按建议修复并复跑 CI。

中风险：

这个问题可能有多个根因。建议先按方案 A 验证，如果仍失败，再进入方案 B。

高风险：

这个问题涉及权限、密钥、生产环境或核心业务逻辑。建议先暂停自动修复，由客户确认边界后再继续。
